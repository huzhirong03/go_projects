# VBA vs Go 程序 性能对比

这里的 `ExtractByKeyword.bas` 是一段等价的 VBA 宏，用来跟「Excel 拆合大师」的"批量提取-单文件"模式做性能对比。

## 行为对齐表

| 维度 | Excel 拆合大师（Go） | VBA 宏（本目录） |
|---|---|---|
| 处理对象 | 单文件模式选中的 xlsx | ActiveWorkbook（你打开的 xlsx） |
| 扫描范围 | 全部 Sheet（除非手动选） | 全部 Sheet |
| 表头 | 第 1 行（默认配置） | 第 1 行（硬编码） |
| 搜索 | 全列搜索 | 全列搜索 |
| 匹配 | 精准 + 包含都勾选 | 包含匹配（InStr） |
| 输出 | 新文件 / 写回源文件新 Sheet | 同左（弹窗选） |
| 保留图片 | 勾选后跟随行 | ❌ 不支持（VBA 复制图片性能很差，干扰对比） |
| 保留样式 | 写回时部分保留 | ❌ 只复制值（最快路径） |

⚠ 不公平的地方：VBA 这版**只复制值**，不带样式、不带图片。所以 VBA 这边其实是"理论最快"的简化实现。如果 Go 程序还能跟它打平甚至超过 → Go 程序的优势更明显。

## 用法

### 方法 A：导入 .bas（推荐）

1. 打开你要测试的 xlsx 文件（双击它）
2. `Alt+F11` 打开 VBA 编辑器
3. 菜单：**文件 → 导入文件** → 选择 `ExtractByKeyword.bas`
4. 按 `F5`（或工具栏运行）执行 `ExtractByKeyword`
5. **弹窗 1**：输入关键词，比如 `汉族`
6. **弹窗 2**：选输出方式
   - **是** → 新文件（会再让你选保存路径）
   - **否** → 写回源文件新 Sheet（完成后记得 Ctrl+S 保存）
7. 完成弹窗显示统计：扫描行数 / 命中行数 / 总耗时

### 方法 B：直接复制粘贴

1. 打开源 xlsx → `Alt+F11`
2. **插入 → 模块** → 把 `ExtractByKeyword.bas` 全文粘进来（包括第 1 行的 Attribute）
3. 按 `F5` 跑

## 公平对比的几个建议

1. **跑同一个文件**：用 Excel 拆合大师跑一遍，再用 VBA 跑一遍，关键词和源文件保持一致
2. **每次都用全新打开的 Excel**：避免 Excel 自己缓存影响第二次结果。最稳：
   - 跑 VBA 前先把 Excel 完全退出再重新打开源文件
   - 跑 Go 程序前确认 Excel 已关掉源文件（避免文件占用）
3. **多跑几次取平均**：第一次启动 Excel/Go 都有冷启动开销，跑 3 次取后两次的平均
4. **关注的指标**：
   - **总耗时**（VBA 弹窗里直接显示）
   - **命中行数**（确认两边匹配语义一致，行数应该一致或极接近）
5. **大文件场景**：10 万行 + 10+ 列才能拉开差距，几千行的小文件 VBA 也很快

## 预期结果

我们的 Go 程序对 100k 行 × 14 列的 xlsx 实测：
- 扫描阶段约 2.7 秒
- 总耗时约 18 秒（包括打开 / 输出 / 图片处理 / 样式克隆）

VBA 在不带图片、不带样式时，纯扫描+写值的耗时通常会更快（因为做的事少），但当源文件有公式、合并单元格、大量样式时，VBA 在 Excel 里"打开"+"渲染"环节就慢了。

## 注意事项

- 跑完后 VBA 输出 Sheet 的**列数 = UsedRange 的列数**。如果你源文件不同 Sheet 的列数不一致，可能会出现"前面 Sheet 的命中行少几列"的现象（因为表头是用第一个有内容的 Sheet 第一行，超过这个列数的数据被截）。这跟 Go 程序按"每个 Sheet 各自表头"的逻辑略有不同。
- VBA 写回模式跑完**不会自动保存**（避免锁文件干扰），需要你手动 Ctrl+S。
- 如果源 xlsx 很大（>200MB），VBA 的 `UsedRange.Value` 可能内存溢出。Go 程序则是流式扫描，不受此限。
